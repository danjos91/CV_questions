### Анализ ответов (Раунд 2)

#### 1. Garbage Collection / Stop-The-World (STW)
**Вопрос:** *В приложении пользователи жалуются на периодические зависания по 2-3 секунды. БД и сеть в порядке. Подозрение на паузы Stop-The-World. 1. Что такое пауза Stop-The-World, почему она происходит и можно ли её избежать? 2. Как бы вы попытались решить эту проблему с длинными паузами в приложении, где критично низкое время отклика?*
*   **Ваш ответ:** STW останавливает все потоки для сборки мусора. Полностью избежать нельзя, но можно настроить. Смотреть логи, исключить БД/сеть. Предложили ZGC.
*   **Фидбек:** `Оценка 7.5/10`. Хороший ответ.
*   **Как улучшить:** Объяснить, почему STW возникает (безопасное перемещение объектов, построение графа достижимости). Упомянуть тюнинг G1: `-XX:MaxGCPauseMillis` (снизить до 50–100 мс). ZGC/Shenandoah — конкурентная сборка, паузы < 1 мс.

#### 2. Transaction Isolation Levels / Lost Update
**Вопрос:** *Финансовое приложение, две операции в рамках одной транзакции: проверка баланса и списание. При быстрых конкурентных запросах баланс становится отрицательным (Lost Update). 1. Какой уровень изоляции стоит в БД по умолчанию и почему он не спасает? 2. Как решить эту проблему (предложите два способа: на уровне БД и на уровне Hibernate)?*
*   **Ваш ответ:** READ COMMITTED по умолчанию, не блокирует конкурентные операции. Решение: атомарная операция в БД; в Hibernate — `@Version` и оптимистическая блокировка.
*   **Фидбек:** `Оценка 8.5/10`. Отличный ответ.
*   **Как улучшить:** Упомянуть пессимистичную блокировку: `SELECT ... FOR UPDATE` или `@Lock(LockModeType.PESSIMISTIC_WRITE)`.

#### 3. Kafka vs RabbitMQ (выбор брокера)
**Вопрос:** *Два Use Case. Case A (Потоковая обработка логов кликов): 5 микросервисов читают миллионы событий, важен порядок, нужна история за весь день. Case B (Очередь задач сжатия картинок): 10 воркеров-консьюмеров, один воркер берет задачу, при падении задача возвращается в очередь. Какой брокер (Kafka/RabbitMQ) выбрать для каждого кейса и почему?*
*   **Ваш ответ:** Case A — Kafka (хранение 24 часа, порядок, миллионы событий/сек). Case B — RabbitMQ (in-memory, retry, ~20k событий/сек).
*   **Фидбек:** `Оценка 9/10`. Очень сильный ответ.
*   **Как улучшить:** Упомянуть Pull vs Push, паттерн Competing Consumers для RabbitMQ.

#### 4. JWT / Logout / Revocation
**Вопрос:** *Для авторизации используется JWT. 1. Опишите процесс аутентификации с помощью JWT. 2. Как реализовать немедленный отзыв токена (revocation) или принудительный Logout, если JWT stateless и токен валиден еще 2 часа? Сравните два подхода.*
*   **Ваш ответ:** Логин/пароль → JWT (header.payload.signature). Blacklist и Token Versioning для отзыва.
*   **Фидбек:** `Оценка 8/10`. Хороший ответ.
*   **Как улучшить:** Упомянуть заголовок `Authorization: Bearer <token>`, компромиссы (Blacklist делает JWT stateful). Короткий Access Token (5–15 мин) + Refresh Token как альтернатива.

#### 5. Cache-Aside vs Write-Through
**Вопрос:** *Разрабатываем систему каталога товаров (частое чтение, редкое обновление). 1. Опишите разницу между стратегиями кэширования Cache-Aside (Lazy Loading) и Write-Through. 2. Как решить проблему неконсистентности, чтобы пользователи не видели старую цену из кэша после ее обновления менеджером?*
*   **Ваш ответ:** Оба при чтении: cache-hit → возврат; cache-miss → БД → кэш → возврат. Lazy Loading пишет в БД и не обновляет кэш; Write-Through пишет в кэш и в БД. Выбрали Lazy Loading.
*   **Фидбек:** `Оценка 7.5/10`. Хороший ответ.
*   **Важное уточнение:** В Cache-Aside при записи нужно **инвалидировать** (удалить) ключ в кэше, а не просто не обновлять. Иначе данные будут устаревать. Флоу: `UPDATE` в БД → `redis.del("product:" + id)` → следующий запрос даст cache-miss и загрузит свежие данные. TTL — дополнительная страховка.
